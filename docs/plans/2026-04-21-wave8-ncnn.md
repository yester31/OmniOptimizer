> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development (권장) 또는 superpowers:executing-plans. Steps are TDD-style (`- [ ]`) — 체크박스가 진행 트래커.

**Goal:** Wave 6이 ORT CPU + OpenVINO로 CPU를 열었고 Wave 7은 archive. Wave 8은 **ncnn (Tencent)** 을 추가해 **모바일/엣지 배포 경로**를 확보. Android/iOS/Raspberry Pi 타겟 지향 — YOLO 생태계에서 ncnn이 사실상 표준이므로 이후 확장의 레버리지가 크다. 본 Wave는 **x86 Intel CPU에서 ncnn을 먼저 벤치**해 OpenVINO/ORT-CPU와 같은 호스트에서 직접 비교한다.

**Architecture:** 변경 지점 4곳. (1) `scripts/_schemas.py::TechniqueSpec.source` Literal에 `ncnn` 추가. (2) `scripts/run_cpu.py`에 ncnn 디스패처 브랜치 + `NcnnAsORT` 어댑터 (Wave 6 `OVRunnerAsORT` 패턴 재사용). (3) `recipes/50–51_*.yaml` 신규. (4) `pyproject.toml` `[cpu]` extras에 `ncnn` 추가 (wheel 가용성 Task 0에서 확인). `run_trt.py` / `run_ort.py` / 기존 Wave 6 코드는 **건드리지 않는다**.

**Tech Stack:** `ncnn` (Python binding, Tencent), `onnx` (ONNX 표현 유지), 변환 툴체인은 Task 0에서 선택:
- (A) `pip install ncnn` wheel의 `ncnn.Net.load_onnx` (존재 시 가장 단순)
- (B) `onnx2ncnn` CLI (ncnn 소스 빌드 or 사전 빌드 배포 다운로드)
- (C) `pnnx` (PyTorch → ncnn, ONNX 우회)

## Scope Boundaries

### In-scope (Wave 8)
- **하드웨어**: x86_64 Intel CPU (AVX2 최소, AVX-512 권장). ARM NEON은 Wave 10+ 타겟 환경 확보 후.
- **모델**: YOLO26n 그대로.
- **정밀도**: FP32 baseline + INT8 PTQ (ncnn 자체 양자화 툴체인).
- **OS**: Linux + Windows (pip wheel 가용 시). macOS best-effort.
- **Vulkan GPU delegate**: **옵션** — host가 Vulkan 지원 GPU + driver 있으면 side-run, 없으면 skip.

### Out-of-scope (future waves)
- **Android/iOS 실기 실행** — 모바일 CI 인프라 필요 (Wave 10+)
- **ARM NEON / SVE** — 테스트 베드 확보 후 (Raspberry Pi 5, Graviton)
- **Apple Silicon Vulkan** — MoltenVK layer 이슈, Wave 11+
- **Vulkan GPU 집중 튜닝** — Wave 8는 X86 CPU가 주 타겟

### Assumptions (착수 전 확인 필요)
- `best_qr.pt` + `qr_barcode.yaml` Wave 6/7 패턴 그대로 사용 가능.
- Windows에서 `pip install ncnn` wheel 제공 여부는 Task 0 spike에서 확인. 부재 시 pre-built binaries 다운로드 또는 WSL 경로로 fallback.

## Known Risks & Decisions

### R1. `onnx2ncnn` / pnnx의 YOLO26n attention 호환성 (Task 0에서 선행 검증)
**Risk**: Wave 3 INC SmoothQuant, Wave 7 torch.export가 모두 YOLO26n attention/Detect head에 막힘. ncnn 변환기도 유사한 패턴 가능. ReduceSum, ScatterND, Slice with dynamic shape 등 attention-specific op에서 converter warning 또는 silent node drop.
**Mitigation**: Task 0 Step 3이 이를 직접 확인. 실패 시 경로 선택:
  - pnnx로 우회 (PyTorch → ncnn 직접, ONNX 스킵)
  - `nodes_to_exclude` 수준의 op 제외 옵션 — ncnn 쪽에서 지원 여부 확인
  - 최악: Wave 8 전체 parked, Wave 9 DirectML로 점프

### R2. Windows pip wheel 부재 가능성
**Risk**: `ncnn` PyPI wheel이 Linux/macOS 위주, Windows 미제공일 수 있음. Wave 7 TFLite가 같은 사유로 block됐음.
**Mitigation**: Task 0 Step 1이 이를 검증. 부재 시:
  - pre-built Windows binary 다운로드 + 환경변수로 경로 지정
  - WSL 사용 (DX 비용)
  - Worst case: Wave 8도 archive, Wave 9 (DirectML — Windows 네이티브)로 점프

### R3. ncnn INT8 accuracy 도메인 mismatch
**Risk**: Wave 6 #33 (`ort_cpu_int8_static`)가 COCO calib + QR eval에서 mAP=0 회귀 발생. ncnn INT8도 같은 파이프라인이므로 동일 위험.
**Mitigation**: Task 7 Step 3에서 실측 확인. mAP=0 재현 시:
  - ncnn `MixPrecision` preset (민감 레이어 FP 유지)
  - Calibration 이미지를 QR 도메인으로 전환 (원본 Wave 5 결정과 배치)
  - 해당 Wave 9+ follow-up으로 이관

### R4. ncnn latency 측정의 thread contract
**Risk**: `ncnn.Net.opt.num_threads` 기본값이 `num_cpu_count()` 즉 hyperthread 포함. Wave 6가 정립한 "물리 코어만 사용" 정책과 어긋남.
**Mitigation**: Task 3에서 `net.opt.num_threads = _resolve_thread_count(recipe)` 명시. 물리 코어 값 강제.
